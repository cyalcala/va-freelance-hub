## 1. EXECUTIVE SUMMARY & WEBSITE PURPOSE
VA.INDEX is a fully autonomous, self-maintaining job feed platform built specifically for Filipino Virtual Assistants.

**The Problem It Solves:** Filipino VAs doing manual job searches on platforms like LinkedIn or Indeed find that 90% of results do not apply due to wrong geography, wrong skillset, or outdated postings. Checking 15+ platforms thoroughly is exhausting.

**The Solution:** A single feed, automatically filtered for Filipino VA relevance, updating every 15-30 minutes continuously. It aggregates opportunities, deduplicates them, ranks them into tiers (Gold, Silver, Bronze), and serves them instantly via a dark-mode obsidian dashboard.

**The Human Stakes:** Every minute the platform is down is a minute a Filipino VA might miss a relevant job opportunity. The site is a living portfolio piece designed to be indestructible and "$0 infrastructure cost always."

## 3. ARCHITECTURE & DATA FLOW CHAIN
The system operates completely decoupled from traditional monolithic backend constraints by leveraging edge SQLite databases and serverless background orchestration.

**The Data Flow Chain:**
1. **Harvesting:** Sources (Reddit `r/VirtualAssistant`, Jobicy, Remotive, HackerNews, RSS feeds like Himalayas/WeWorkRemotely, JSON feeds like JobStreet) are polled every 15-30 minutes.
2. **Triaging & Sifting:** The Zig Native Engine (The "Titanium Sieve") kills Tech/Exec/Blog noise with near-zero latency. TS Jaro-Winkler serves as a cloud fallback.
3. **Filtering & Deduplication:** Geographic EMEA/restriction filters applied. Cryptographic/Semantic deduplication using a `(title, company)` composite fingerprint prevents hash explosion.
4. **Tier Scoring:**
   - **GOLD (Tier 1):** Explicitly mentions Philippines/Filipino/SEA. High priority.
   - **SILVER (Tier 2):** Globally accessible, no geographic restrictions.
   - **BRONZE (Tier 3):** Accessible but lower relevance signal.
5. **Storage:** Written to Turso DB via `@libsql/client/http` (to prevent native binary resolution errors). Drizzle ORM manages the schema.
6. **Delivery:** Vercel API routes (Astro SSR) serve the data to the frontend using Edge-Side Stale-While-Revalidate caching, providing <100ms response times.

### ARCHITECTURE.md
# VA.INDEX Architecture & Evolution (v5.2)
*A Deep Technical Breakdown of the Autonomous Signal-Capture Platform*

## 1. The Core Paradigm & Topology
The VA.INDEX platform is designed as an indestructible, "set-it-and-forget-it" pipeline. It operates completely decoupled from traditional monolithic backend constraints by leveraging edge SQLite databases, serverless background orchestration, and a statically generated frontend.

The system is highly abstracted. Its core intelligence, scraping arrays, and scam heuristics are externalized into a modular `@va-hub/config` package, enabling the entire codebase to be instantly cloned and mapped to any niche (e.g., Web3 Developers, Cybersecurity, UI/UX Designers) without rewriting business logic.

## 2. Infrastructure & Compute Layers
### Database Storage (Turso + LibSQL Edge)
- **Engine:** LibSQL (SQLite fork optimized for low latency edge replication).
- **ORM Interface:** **Drizzle ORM** is utilized for strict, typesafe SQL query building and schematic migrations.
- **Relational Schema:**
  - `agencies`: Stores top-level entity metadata.
  - `opportunities`: Stores individual job signals extracted dynamically. Contains a cryptographically unique `content_hash` and an indexed `latest_activity_ms` column for hardware-accelerated sorting.

### The Titanium Sieve (Zig Native Engine)
- **The Sifter (`sifter.zig`):** A high-performance, SIMD-ready binary module that hard-kills Tech/Exec/Blog noise at the hardware level with near-zero latency (~7.9ms per batch).
- **Bun FFI Bridge:** Native interoperability allows TypeScript to call Zig functions with zero overhead.

### Orchestration Execution (Trigger.dev v3)
- **Engine Shift:** We explicitly operate on Trigger.dev's V3 architecture. Traditional Vercel Serverless Functions enforce a brutal 10-60 second execution timeout limit, making heavy scraping unviable. V3 background workers pull the compute off the main thread entirely.
- **Strategic Synchronization:** All harvest tasks populate the `latest_activity_ms` field [Math.max(posted, scraped)]. This allows the frontend to query the DB using a composite index `(tier, latest_activity_ms)`, dropping sorted response times from >500ms to <100ms.

## 3. The Backend Service Workers (Background Jobs)
All background operations are highly surgical, enforcing O(1) growth caps and strict deduplication constraints.

### The Harvester (`scrape-opportunities.ts`)
- **Cron Cycle:** `0 */2 * * *` (Every 120 minutes).
- **RSS Parser Vectors:** Rapidly fetches and extracts payload bodies from Himalayas, WeWorkRemotely, RemoteOK, and ProBlogger.
- **REST Protocol Vectors:** Directly pings the Jobicy API, HackerNews, and Reddit JSON pipelines.
- **Cryptographic Deduplication:** Concatenates `title + company + source_platform` into a unified hash array.

### The Scam Shield (`trust.ts`)
- **RegEx Filtration:** Employs advanced `/(\$?\d+\s*\/.*day)/i` structural regex matching directly onto incoming description payloads to catch scams.

### The Phantom Link Verifier (`verify-links.ts`)
- **Cron Cycle:** `0 6 * * *` (Daily at 6:00 AM UTC).
- **ATS Semantic Checking:** Instead of just checking HTTP `404 Not Found` statuses, it fetches HTML to look for ATS ghost-job signatures.

### The Resilience Watchdog (`resilience-watchdog.ts`)
- **Cron Cycle:** `0 */6 * * *` (Every 6 hours).
- **Pulse Audit:** Detects 'Silent Blackouts' if no data has been discovered in 4 hours.

## 4. Frontend Rendering (Unified SSR)
- **Engine Priority:** Both **Astro** (`apps/frontend`) and **Next.js** (`apps/web`) configured for **Strict SSR** (output: 'server').
- **Cache-Control Shields:** Leverage **Edge-Side Stale-While-Revalidate (SWR)** caching (`s-maxage=60, stale-while-revalidate=30`).
- **Aesthetic Refinement:** O(1) performance focus with Tailwind CSS and zero-JS hydration (Astro).

## 5. The Failsafe Subsystem
- **Snapshot CLI (`bun run save`)**: Generates an alphanumeric mythical ID format (`Hydra_2026-03-XX`).
- **Restoration Hook (`bun run restore`)**: Dynamically rolls Git memory variables back through history.

## 6. The Agentic SRE Sentinel
The platform is protected by an autonomous "Agentic" sentinel powered by Gemini 1.5 Flash.
- **Zero-Cost Enforcement**: A strict quota of 10 AI calls/day is tracked in `.sentinel-quota.json`.
