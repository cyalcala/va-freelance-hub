## 1. EXECUTIVE SUMMARY & WEBSITE PURPOSE
VA.INDEX is a fully autonomous, self-maintaining job feed platform built specifically for Filipino Virtual Assistants.

**The Problem It Solves:** Filipino VAs doing manual job searches on platforms like LinkedIn or Indeed find that 90% of results do not apply due to wrong geography, wrong skillset, or outdated postings. Checking 15+ platforms thoroughly is exhausting.

**The Solution:** A single feed, automatically filtered for Filipino VA relevance, updating every 15-30 minutes continuously. It aggregates opportunities, deduplicates them, ranks them into tiers (Gold, Silver, Bronze), and serves them instantly via a dark-mode obsidian dashboard.

**The Human Stakes:** Every minute the platform is down is a minute a Filipino VA might miss a relevant job opportunity. The site is a living portfolio piece designed to be indestructible and "$0 infrastructure cost always."

## 3. ARCHITECTURE & DATA FLOW CHAIN
The system operates completely decoupled from traditional monolithic backend constraints by leveraging edge SQLite databases and serverless background orchestration.

**The Data Flow Chain:**
1. **Harvesting:** Sources (Reddit `r/VirtualAssistant`, Jobicy, Remotive, HackerNews, RSS feeds like Himalayas/WeWorkRemotely, JSON feeds like JobStreet) are polled every 15-30 minutes.
2. **Triaging & Sifting:** The Zig Native Engine (The "Titanium Sieve") kills Tech/Exec/Blog noise with near-zero latency. TS Jaro-Winkler serves as a cloud fallback.
3. **Filtering & Deduplication:** Geographic EMEA/restriction filters applied. Cryptographic/Semantic deduplication using a `(title, company)` composite fingerprint prevents hash explosion.
4. **Tier Scoring:**
   - **GOLD (Tier 1):** Explicitly mentions Philippines/Filipino/SEA. High priority.
   - **SILVER (Tier 2):** Globally accessible, no geographic restrictions.
   - **BRONZE (Tier 3):** Accessible but lower relevance signal.
5. **Storage:** Written to Turso DB via `@libsql/client/http` (to prevent native binary resolution errors). Drizzle ORM manages the schema.
6. **Delivery:** Vercel API routes (Astro SSR) serve the data to the frontend using Edge-Side Stale-While-Revalidate caching, providing <100ms response times.

## 5. DATABASE SCHEMA (`packages/db/schema.ts`)
1. **opportunities**:
   - `id`, `title`, `company`, `type`
   - `sourceUrl`, `sourcePlatform`, `tags`, `payRange`, `description`
   - `postedAt` (EVENT_TIME)
   - `scrapedAt` (PROCESSING_TIME/Heartbeat)
   - `createdAt` (INGESTION_TIME)
   - `isActive` (Boolean)
   - `tier` (1=Gold, 2=Silver, 3=Bronze, 4=Trash)
   - `latestActivityMs` (Composite index target)
   - Unique Index on `(title, company)` to enforce semantic deduplication.
2. **agencies**: Directory of VA agencies, their status, hiring heat, friction level.
3. **system_health**: Tracks status of scrapers (OK/FAIL) and last success timestamps.
4. **vitals**: Tracks AI quota limits, lock status, recovery events.
5. **logs**: Centralized logging table.

## 6. KEY MECHANICS & INVIOLABLE RULES
1. **The Four Timestamp Axes**:
   - `posted_at`: Ground truth age of the job. Used for feed ordering.
   - `created_at`: Ingestion time. Used to measure pipeline throughput.
   - `scraped_at`: Processing time heartbeat. Does not equal data freshness.
2. **LibSQL Async Factory**: Tasks must use `const { client, db } = await createDb();` dynamically inside run cycles. Static imports break native binary resolutions on Trigger.dev.
3. **Vercel Config is Sacred**: `vercel.json` dictates the deployment. Never run Vercel CLI from the root directory; it lives in `apps/frontend`.
4. **Semantic Deduplication**: Instead of URL hashing, dedup is performed via `(title, company)`. This fixes the "Hash Explosion" where platforms like RemoteOK rotate URLs for the same job.
5. **Zombie Horizon**: Jobs older than 14-21 days (`posted_at`) are automatically deactivated.
6. **Cloud vs Local**: Development is typically Windows, Deploy is Linux (Vercel/Trigger.dev). Local Windows symlink/binary errors do not block pushing to GitHub.

## 7. DEPLOYMENT PROTOCOL
* **Environment Variables** exist across 3 distinct zones, do not share them locally:
  1. Local `.env`
  2. Vercel Dashboard (Frontend/DB tokens)
  3. Trigger.dev Dashboard (Background workers)
* **Trigger.dev Deployments**: Never deploy from local Windows (`npx trigger.dev deploy`). Push to GitHub, where GitHub Actions (`.github/workflows/trigger-deploy.yml`) handles the deployment natively on Linux.
* **Health Monitoring**: System health is ALWAYS checked at `https://va-freelance-hub-web.vercel.app/api/health`. Look for `status: "HEALTHY"`, `stalenessHrs < 1`, and `totalActive` growth.
